## Twitter / X — technical thread (2-tweet follow-up)

```
1/ how we did it.

v6.0's only documented drift failure mode was arg_swap — when the
model produces right arg names but wrong values per slot. all 22
features passed. schema checks pass. prompt-overlap passes.

we needed a feature none of them encoded: positional order.
```

```
2/ for each arg-pair (ki, kj) where both call values have a
detectable position in the prompt:
  schema order: schema_order[ki] < schema_order[kj]
  prompt order: pos(call_args[ki]) < pos(call_args[kj])
  inverted if the two disagree.

arg_order_inversion = inversions / eligible_pairs.

arg_swap lifts +0.091. no training on arg_swap specifically —
it emerged from one extra feature fed into the same calibrated LR.
```

## LessWrong / Alignment Forum — short-form (comment-level)

```
styxx v6.1 shipped: tool-call drift detector retrained with one new
feature (arg_order_inversion). closes ~25% of the arg_swap gap
(0.66 → 0.76, overall AUC 0.92 → 0.94).

interesting because arg_swap is the case where all schema/prompt-
overlap features pass by construction — the swap is invisible to
any feature that treats call args as a set. it only becomes visible
when you look at positional ordering between prompt values and
schema slot order.

arg_swap is not fully solved — remaining gap is where swapped
values share a prompt position (numerical ambiguity, e.g.
"divide 5 by 5") or one value was synthesized (not in prompt).
fully closing it needs embedding-based per-slot semantic fit,
scoped for v3.

transparency note: this was a targeted fix on a previously
documented failure mode, not a blind-alley retrain. v6.0 published
the 0.66 number openly in CHANGELOG; v6.1 publishes the 0.76 + the
remaining gap. we'll keep doing that.
```
